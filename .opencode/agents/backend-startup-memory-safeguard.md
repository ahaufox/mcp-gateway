# 启动与内存防范规则 (Startup & Memory Safeguard)

## 1. 异步启动死锁防范
近期由于数据库懒加载设计与同步方法的混用，曾出现服务启动挂起的问题（见 `walkthrough.md`）。因此：
- **禁止在模块级别执行阻塞操作**: 严禁在模块导入阶段初始化建立数据库连接或执行耗时的索引同步。
- **强制懒加载与生命周期托管**: 数据库引擎 `get_engine()` 以及模型的加载，必须由 FastAPI 的 `lifespan` 托管。
- **后台任务隔离**: 耗时任务（如大型向量索引构建）如果在启动时触发，必须包装在后台异步任务中，不得阻塞主线程的启动，确保 API ` /health` 尽快响应。

## 2. 内存与 GPU 防范
- **分步加载策略**: 当集成多种大型 AI 模型（如 ASR 和 Rerank 模型）时，不要并发加载。必须控制先后顺序以避免短时间内显存尖峰（Spike）导致 OOM。
- **配置隔离**: 模型路径与缓存策略必须能够通过配置文件控制开启/关闭。如果环境受限，系统必须能够平稳降级至“无模型”模式，而不是直接 Crash。
- **进程资源清理**: 任何处理长连接或多进程任务的代码，遇到异常时必须具备 `finally` 块显式释放资源。

## 3. 网络与代理兼容性
- 代码在初始化阶段若涉及代理配置清理（例如动态消除代理环境变量以适配特殊库），必须理清与系统 Logger 的初始化时序，防止时序冲突导致丢失启动日志。
