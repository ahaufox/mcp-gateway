## 1. Overview
随着 Model Context Protocol (MCP) 的普及，设计师不再局限于传统的 "Pixel Pushing"，而是可以通过 MCP 将 AI Agent 深度集成到 **设计系统管理**、**UI 代码生成**、**可访问性测试 (A11y)** 和 **视觉 QA** 流程中。

"设计师必备 MCP" 指的是能让 AI 助手直接理解设计上下文（Figma 图层、Token、配色）、操作设计工具或验证设计产出的协议服务端。

## 3. Strengths & Weaknesses

### 🎨 Figma MCP (官方/社区版)
*   **Strengths**:
    *   **Single Source of Truth**: 直接读取设计源文件，生成的代码准确对应 Figma Auto Layout 结构。
    *   **Token Awareness**: 能识别 Variables (如 `primary-500`) 而非硬编码 Hex 值。
*   **Weaknesses**:
    *   **Read-Only 限制**: 大多数实现目前主要用于读取图层信息生成代码，反向修改 Figma 能力较弱。
    *   **性能消耗**: 解析大型 Figma 文件可能较慢。

### 🧵 Stitch MCP (Google Stitch)
*   **Strengths**:
    *   **Design DNA**: 独有的“视觉提取”能力，能模仿现有截图的风格生成新界面。
    *   **闭环能力**: 支持从生成到代码导出的全流程。
*   **Weaknesses**:
    *   **生态封闭**: 主要围绕 Stitch 平台生态，对非 Stitch 项目的支持可能有限。

### ♿ A11y MCP
*   **Strengths**:
    *   **专业性**: 专注于 WCAG 标准，比通用 LLM 视觉评估更准确、合规。
*   **Weaknesses**:
    *   **单一功能**: 仅做检查，无法自动修复设计文件。

## 4. Selection Advice (选型建议)

*   **场景 A：你是“全栈设计师” (Design Engineer)**
    *   **必选**: **Figma MCP** + **Filesystem MCP**。
    *   **理由**: 你需要让 AI 读取 Figma 并直接在本地项目生成 React/Tailwind 代码。

*   **场景 B：你是“产品设计师” (Product Designer)**
    *   **必选**: **Stitch MCP**。
    *   **理由**: 快速通过文字生成高保真原型，进行方案验证和风格探索。

*   **场景 C：你是“设计系统负责人” (Design Ops)**
    *   **必选**: **Figma MCP** + **A11y MCP**。
    *   **理由**: 你需要自动化检查组件库的色值规范和对比度合规性。

## 5. Ecosystem & Next Steps

目前设计类 MCP 处于爆发初期。除了上述核心工具，建议关注以下集成方向：
*   **Storybook MCP**: 将设计组件与已开发组件进行自动化匹配。
*   **Color Palette Agents**: 专门用于生成和谐配色并导出 JSON Token 的轻量级 Server。
